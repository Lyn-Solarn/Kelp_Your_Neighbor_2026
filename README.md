# Marinote

Marinote is a React + Vite social posting app for tidepool observations. Users can browse posts, add sightings, leave comments, and manage their own posts.

This refresh focuses on making the original project more production-ready without changing the core concept:

- safer Supabase bootstrapping with blank credentials supported
- cleaner layout and stronger page hierarchy
- improved loading, empty, and error states
- local browser session handling instead of a database-wide `current_user` flag
- a dedicated `/demo` route with seeded mock content for design review

## Local setup

1. Install dependencies:

```bash
npm install
```

2. Create `.env.local` from the example file:

```bash
cp .env.example .env.local
```

3. Add your environment values to `.env.local`:

```bash
VITE_SUPABASE_URL=
VITE_SUPABASE_ANON_KEY=
VITE_NEWS_API_KEY=
```

`VITE_NEWS_API_KEY` is used by the `/ocean-news` page to load top headlines from NewsAPI using
the `top-headlines` endpoint with `category=science`, `q=ocean`, and `pageSize=3`.

4. Start the app:

```bash
npm run dev
```

## Demo route

If you want to review the updated design before wiring the database, open:

```text
/demo
```

This route does not require Supabase credentials.

## Suggested Supabase schema

The current code expects these tables:

### `user`

- `id` bigint primary key generated by default
- `username` text unique not null
- `secret_code` text not null
- `created_at` timestamp with time zone default now()

### `post`

- `id` bigint primary key generated by default
- `posted_by` bigint references `user(id)` not null
- `title` text not null
- `description` text
- `image` text
- `pearls` integer default 0
- `created_at` timestamp with time zone default now()

### `comment`

- `id` bigint primary key generated by default
- `posted_by` bigint references `user(id)` not null
- `og_post` bigint references `post(id)` not null
- `text` text not null
- `created_at` timestamp with time zone default now()

## Important note on auth

The app still uses pseudo-auth based on a username and secret code stored in the database. That is acceptable for a prototype, but not for a true production launch. For production, plan to migrate to Supabase Auth or another managed auth provider before collecting real user accounts.
